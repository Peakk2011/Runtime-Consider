The New application
Runtime Consider (Codename)

Type: Desktop App (but Web Mobile is optional)
Concept MVP Planning

เอกสารนี้เป็นข้อกำหนดข้อความธรรมดาๆที่ละเอียดหน่อย
ตั้งใจเขียนให้เป็นข้อความต่อเนื่อง ไม่ใช่แบบหัวข้อย่อย

มันต้องยังคงอ่านได้แม้ผ่านไปหลายเดือนหรือหลายปีโดยไม่ต้องอาศัยบริบทจากความทรงจำ หากการใช้งานจริงไม่สอดคล้องกับเอกสารนี้ แสดงว่าการใช้งานจริงนั้นผิด

1.

Runtime Consider เป็นแอปพลิเคชันบนเดสก์ท็อปที่สร้างขึ้นเพื่อบันทึกความเป็นจริง ไม่ใช่เพื่อวางแผนในแต่ละวันของคุณเมื่อสร้างสิ่งต่างๆ
แนวคิดหลักที่เรียบง่ายแต่ลึกมากๆคือ: แต่ละวันในชีวิตของมนุษย์ถือเป็นการดำเนินการแบบ Realtime และสิ่งที่เกิดขึ้นระหว่างการดำเนินการนั้นสามารถบันทึกได้เพียงครั้งเดียวเท่านั้น แอปนี้ไม่ได้พยายามปรับปรุงผู้ใช้ กระตุ้นผู้ใช้ หรือเพิ่มประสิทธิภาพพฤติกรรมอะไรทั้งสิ้นเพราะจะต้องมีอยู่เพียงเพื่อสะท้อนสิ่งที่เกิดขึ้นจริง
เครื่องมือเพิ่มประสิทธิภาพการทำงานส่วนใหญ่ทำงานในอนาคต พวกมันถามว่าพวกคุณว่าต้องการทำอะไร คุณควรทำอะไร หรือคุณหวังว่าจะทำอะไรให้สำเร็จแต่
Runtime Consider ตั้งใจจะปฏิเสธกรอบความคิดนั้น แอปนี้ทำงานในอดีตทั้งหมด มันต้องการจะสนใจในเฉพาะสิ่งที่เกิดขึ้นแล้วเท่านั้น สิ่งใดก็ตามที่ไม่ได้เกิดขึ้นจะไม่ถือว่าเป็นความล้มเหลวหรือช่องว่าง มันเป็นเพียงการไม่มีข้อมูล
โมเดลทางความคิดสำหรับแอปพลิเคชันนี้ไม่ใช่รายการสิ่งที่ต้องทำ บันทึกประจำวัน หรือตัวติดตามนิสัย โมเดลทางความคิดที่ถูกต้องนั้นใกล้เคียงกับบันทึกระบบหรือประวัติการเปลี่ยนแปลงในระบบควบคุม Version มากกว่า (GIT) แต่ละวันเป็นบันทึกที่เพิ่มเข้าไปได้เท่านั้น เมื่อเขียนแล้วจะไม่สามารถเปลี่ยนแปลงได้ ไม่มีแนวคิดเรื่องการแก้ไข การปรับปรุง หรือการเพิ่มประสิทธิภาพของอดีต อดีตนั้นไม่สามารถเปลี่ยนแปลงได้ตามการออกแบบ
ความไม่สามารถเปลี่ยนแปลงนี้ไม่ใช่ข้อจำกัดทางเทคนิค แต่มันคือการตัดสินใจด้าน UX ที่สำคัญ การอนุญาตให้แก้ไขจะเปลี่ยนแอปให้กลายเป็นเครื่องมือเล่าเรื่อง ที่ผู้ใช้สามารถปรับเปลี่ยนความทรงจำเพื่อให้รู้สึกดีขึ้นและมันไม่ใช้หลักความเป็นจริงที่แอพของเราเช่น Runtime Consider จะดำรงอยู่และไม่ใช่จุดยืนของพวกเรา ที่พวกเราทำได้นั้นคือปฏิเสธที่จะทำเช่นนั้น มันถือว่าความจริงมีค่ามากกว่าความสะดวกสบาย

2.

เมื่อเปิดแอปนี้ถูกเปิดขึ้นมา อย่างแรกที่มันจะทำคือ มันจะตรวจสอบเงื่อนไขเดียวว่า: วันนี้ได้รับการบันทึกแล้วหรือยัง?
หากคำตอบคือไม่ใช่ แอปพลิเคชันจะแสดงพื้นที่สำหรับเขียนทันที เมื่อคุณโฟกัสในพื้นที่สำหรับการเขียนอยู่แล้ว ไม่จำเป็นต้องนำทาง อินเทอร์เฟซ (Interface) ทำให้ชัดเจนว่าวันนี้ยังไม่ได้รับการแก้ไข
ถ้าคำตอบคือใช่ พื้นที่เขียนจะถูกล็อกทั้งทางสายตาและทางฟังก์ชัน และแอปพลิเคชันจะเปลี่ยนไปเน้นที่ประวัติการใช้งานแทน

พื้นที่เขียนนั้นถูกออกแบบมาให้เรียบง่ายที่สุด เป็นเพียงช่องป้อนข้อความหลายบรรทัดธรรมดา ไม่มีรูปแบบการจัดวาง ไม่มีมาร์กดาวน์ (Markdown) ไม่มีแถบเครื่องมือ และไม่มี UI ที่ตกแต่งสวยงาม แต่จะเป็นอีกรูปแบบอักษรเป็นแบบตัวอักษรคงที่และดูเหมาะสม ไม่สวยเกินไป ไม่น่าเกลียดเกินไป แต่ทุกจุดถูกออกแบบมาให้ตั้งอยู่ในจุดนั้น สำหรับ UI แบบนี้ไม่ใช่เพื่อความสวยงามแบบย้อนยุค แต่เพื่อเพิ่มความรู้สึกเหมือนกำลังบันทึกเรื่องราวมากกว่าการแต่งบทความ แต่ละบรรทัดถือเป็นข้อเท็จจริง การกด Enter จะสร้างบรรทัดใหม่ การกระทำเพียงครั้งเดียวจะบันทึกข้อมูลของวันนั้นๆ
การบันทึกข้อมูลของวันนั้นๆ ไม่สามารถย้อนกลับได้ เมื่อมีการกดบันทึกแล้ว ช่องป้อนข้อมูลจะกลายเป็นแบบอ่านอย่างเดียวและจะเป็นเช่นนั้นตลอดไป ไม่มีการยกเลิก ไม่มีโหมดแก้ไข และไม่มีทางออกฉุกเฉินใดๆ นี่เลียนแบบพฤติกรรมของระบบระดับต่ำ (Low level) ที่เป็นการเขียนลงดิสก์เป็นการทำงานแบบทางเดียว ความร้ายแรงของการกระทำนี้เป็นไปโดยเจตนา แอปควรจะให้ความรู้สึกอึดอัดเล็กน้อยในขณะนี้
ด้านล่างพื้นที่เขียนคือมุมมองประวัติการใช้งาน ประวัติการใช้งานเป็นแบบเส้นตรง เลื่อนดูได้ และอ่านได้อย่างเดียว วันที่เรียงลำดับจากใหม่ที่สุดไปเก่าที่สุด ไม่มีตารางปฏิทิน ไม่มีภาพรวมรายเดือน และไม่มีการแสดงภาพเวลาแบบนามธรรม เวลาไหลไปในทิศทางเดียวเท่านั้น การคลิกที่วันในอดีตจะไม่เปิดโปรแกรมแก้ไข แต่จะแสดงวันนั้นขึ้นมาให้เห็นเพื่ออ่านเท่านั้น
หากมีวันใดที่ไม่มีข้อมูล แอปจะแสดงให้เห็นอย่างชัดเจน วันที่ว่างเปล่าจะไม่ถูกเติมข้อมูลอัตโนมัติ ซ่อน หรือปรับแต่ง ข้อมูลที่ว่างเปล่าถือเป็นข้อมูลที่ถูกต้อง มันจะไม่ได้พยายามคาดเดาอารมณ์ ประสิทธิภาพ หรือเจตนาจากข้อมูลที่หายไป
ในด้านการออกแบบ แอปนี้มีความทันสมัยและเรียบง่าย ภาษาการออกแบบยืมแนวคิดจากเทอร์มินัล (Terminal) และโปรแกรมแก้ไขโค้ด แต่หลีกเลี่ยงการเลียนแบบทางภาพ ไม่มีสีนีออน ไม่มีลูกเล่นใดๆ การจัดวางตัวอักษรและระยะห่างทำหน้าที่ส่วนใหญ่ อินเทอร์เฟซควรให้ความรู้สึกสงบ แม่นยำ และจริงจัง ไม่ควรให้ความรู้สึกสนุกสนานหรือกระตุ้น

3.

แผนงานสำหรับ Runtime Consider นั้นสั้นโดยเจตนา ระยะแรกคือการสร้างความชัดเจนในเชิงแนวคิด: การกำหนดปรัชญา ฟีเจอร์ที่ไม่จำเป็น และข้อจำกัด ระยะต่อไปคือ MVP (Minimum Viable Product) ที่มีเพียงพื้นที่จัดเก็บข้อมูลในเครื่อง ข้อมูลป้อนเข้าประจำวัน การบันทึกการเปลี่ยนแปลง และมุมมองประวัติ หากใช้เวลานานกว่า 4 วันที่มุ่งเน้น ขอบเขตงานก็จะเปลี่ยนไปแล้ว
ความสำเร็จของแอปพลิเคชันนี้ไม่ได้วัดจากจำนวนผู้ใช้ที่เพิ่มขึ้น ตัวชี้วัดการมีส่วนร่วม หรือการรักษาผู้ใช้ไว้ ความสำเร็จอยู่ที่การคงความซื่อสัตย์หลังจากใช้งานไปหลายเดือน จะถือว่าประสบความสำเร็จหากมันไม่ถูกเปลี่ยนให้กลายเป็นรายการสิ่งที่ต้องทำ และจะถือว่าประสบความสำเร็จหากการอ่านบันทึกเก่าๆ ให้ความรู้สึกว่าเป็นข้อเท็จจริงมากกว่าการคัดสรร
มีอยู่กฎสุดท้ายที่เหนือกว่าการอภิปรายเกี่ยวกับคุณสมบัติทั้งหมด หากเมื่อใดก็ตามที่เกิดคำถามว่า “เราควรเพิ่มสิ่งนี้เพื่อช่วยเหลือผู้ใช้หรือไม่” คำตอบคือไม่ Runtime Consider ไม่ได้ช่วย แต่เป็นการสะท้อนกลับ ข้อจำกัดนั้นคือตัวผลิตภัณฑ์เอง
